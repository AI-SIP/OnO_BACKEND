# OnO Analysis

## Workflow

1. 인자가 없으면 `git status`와 `git diff HEAD` 기준으로 현재 변경사항을 분석 대상으로 삼는다.
2. 증상, 원하는 결과, 관련 파일/로그가 불명확하고 현재 변경사항에서도 분석 대상이 좁혀지지 않으면 먼저 짧게 확인한다.
3. 가능한 원인 가설을 3~5개로 나눈다.
4. 관련 코드 흐름, 설정, 로그, git diff/log를 읽기 전용으로 조사한다.
5. 각 가설을 `파일:라인` 근거로 지지하거나 반박한다.
6. 1차 결론을 스스로 반증한다. 상위 레이어에서 막히는지, 실제 호출 가능한지, 다른 설정이 우선하는지 확인한다.
7. 완료 직전에 아래 자가 반증 질문에 답하고, 필요한 경우 결론을 수정한다.
   - 내가 놓쳤을 가능성이 큰 것은?
   - 이 변경이 실제 사용자에게 깨질 수 있는 경로는?
   - 이 결론이 틀렸다면 어떤 조건 때문인가?
8. 코드는 수정하지 않는다.

## Output

- 한 문장 요약
- 근거 목록
- 근본 원인과 확신도
- 자가 반증 결과
- 다음 단계 제안
