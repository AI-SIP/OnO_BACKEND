# OnO Plan

## Workflow

1. 인자가 없으면 `git status`와 `git diff HEAD` 기준으로 현재 변경사항의 목적과 영향 범위를 먼저 파악한다.
2. 요청 기능 또는 현재 변경사항의 사용자 시나리오와 운영 영향 범위를 정리한다.
3. 관련 코드, API 계약, DB schema, 인증/권한, 상태 흐름, 기존 문서를 확인한다.
4. 코드는 구현하지 않는다. 문서만 작성한다.
5. `/Users/ksm/programing/sw_maestro/OnO_BACKEND/backend/docs` 아래에 기능을 알아볼 수 있는 새 폴더를 만든다.
6. 구현 명세서에는 다음 항목을 포함한다.
   - 기능 목적과 사용자 시나리오
   - 구현 범위와 제외 범위
   - 관련 화면, 라우팅, 상태 관리, API 영향
   - 데이터 흐름과 에러 처리
   - 구현 단계
   - 검증 방법
   - 배포 리스크와 확인 질문
7. 이후 `feat` 요청에서 바로 구현할 수 있을 정도로 파일, 클래스, 테스트 방향을 구체화한다.

## OnO 기준

- 실사용자 데이터, 학습 기록, 인증, 결제, 동기화, 알림, API 호환성을 최우선으로 본다.
- DB schema 변경이 필요하면 `ddl-auto=create` 또는 `create-drop`을 제안하지 않는다.
- 불확실한 API 계약이나 null 가능성은 추측하지 말고 확인 질문으로 남긴다.
