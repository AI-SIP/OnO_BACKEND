# /plan — 구현 계획서

**읽기 전용 명령: 코드 수정·커밋 절대 금지. 계획서 파일만 저장.**

## 작업: $ARGUMENTS

## 단계

1. **조사** — `code-tracer` 에이전트로 관련 코드 흐름 파악. 필요 시 `permission-auditor`, `perf-reviewer`, `db-safety` 에이전트 추가 투입. 근거는 `파일:라인`.  
   - **DB 스키마 변경이 포함되면**: `migration-safety` 에이전트로 마이그레이션 안전성 사전 검토.  
   - **DTO·엔드포인트 변경이 포함되면**: `api-contract-checker` 에이전트로 Flutter 앱 breaking change 여부 확인.

2. **계획서 작성** — 아래 섹션을 포함:
   - **목적**: 한 줄 요약
   - **영향 범위**: 수정될 파일·패키지·레이어 목록
   - **구현 단계**: 번호 있는 순서 (각 단계 독립적으로 구현 가능하게)
   - **위험 요소**: FCM 발송·데이터 마이그레이션·권한 변경 등 주의 지점
   - **테스트 방향**: 어떤 경우를 검증해야 하는가 (사용자 격리 포함)
   - **커밋 분할 제안**: 논리 단위별 커밋 메시지 초안

3. **저장** — 계획서를 `docs/plans/<작업명>_plan.md`에 저장. 저장 경로를 사용자에게 알린다.

4. **승인 대기** — "계획서를 검토해주세요. 승인하면 /feat으로 진행합니다." 라고 안내.

> 🔴 계획서 검토는 가장 중요한 체크포인트다. 사용자 확인 전 구현 시작 금지.
